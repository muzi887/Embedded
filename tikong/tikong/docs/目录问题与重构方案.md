## 第一部分：当前文件目录有什么问题

### 1.1 现状树（摘要）

```text
tikong/                          ← 工程根
├── USER/                        入口、中断、Keil、部分流程图 md
├── SYSTEM/                      delay / sys / usart（驱动）
├── drivers/                     片外器件
├── board/                       板级（与 drivers 边界模糊）
├── app/                         业务（comm / utils 再拆）
├── third_party/                 cJSON、sha1
├── CORE/                        启动与 CMSIS
├── FWLIB/                       标准外设库
├── docs/                        部分专文
├── OBJ/、.cache/                编译产物
├── PROJECT_STRUCTURE.md、FUNCTION_SUMMARY.md、
│   parseSerialData_ANALYSIS.md、CURSOR_JUMP_GUIDE.md、
│   FIX_JUMP_ISSUE.md、IDE_INDEX.md、snprintf_USAGE.md …
├── AT.txt、开发主要问题.doc、梯控门禁网关V2_….docx …
└── fix_code_index.*、compile_commands.json …
```

与你已熟悉的 **江协 F103 工程**对比：

| 江协习惯 | 本工程现状 |
| --- | --- |
| `User` / `System` / `Hardware` / `Library` / `Start` | `USER` / `SYSTEM` / `drivers`+`board` / `FWLIB` / `CORE` |
| 硬件模块集中在 `Hardware` | 器件拆在 `drivers` 与 `board` 两处 |
| 业务通常很薄或就在 `User` | 业务在 `app`，但与通道、存储、云端搅在一起 |
| 说明文档不进源码根或很少 | 根目录堆满分析/IDE md 与产品 docx |

### 1.2 问题清单（为什么「不够清晰」）

#### 问题 A：外层命名与江协课不一致

- 打开本仓库无法立刻对应「User / System / Hardware / Library / Start」。
- `CORE`、`FWLIB`、`USER`、`SYSTEM` 大小写与习惯混用，和入门工程对照成本高。

#### 问题 B：硬件被拆成 `drivers` + `board`，边界不清

- 例如：RTC 芯片在 `drivers`，时间抽象在 `board/RTC`；EEPROM/IIC 在 `drivers`；限层、HC595、timer 在 `board`。
- 新人或自己过一段时间后，会反复问：「这个该进 drivers 还是 board？」
- 江协风格下，这些本应都落在 **`Hardware`**（内部再分子目录即可）。

#### 问题 C：串口驱动与业务曾长期纠缠，目录上仍易混

- 当前约定已是：`SYSTEM/usart` = 驱动，`app/comm` = `*Process` 业务——这是对的。
- 但 `SYSTEM` 名字像「系统层」，里面却塞了五路 USART；与江协「Serial 进 Hardware」的直觉不一致。
- 文档与旧习惯里仍可能出现 `HARDWARE/`、`QRProcess` 单入口等过时说法，和真实路径打架。

#### 问题 D：业务模块没有按领域拆开

现有 `app/` 大致是：

| 位置 | 实际职责 | 问题 |
| --- | --- | --- |
| `app/comm/qr_comm` | 通行读头 | 与 4G、PC、485 同级，看不出「通行主路径」 |
| `app/utils/cmd`、`data_handler` | 权限/验签/帧解析 | `utils` 名过泛，不像业务模块 |
| `app/comm/g4g_comm`、`Live_data*` | 云端 JSON | 通道文件与协议文件分散 |
| `app/blackList` | 黑名单存储业务 | 与 EEPROM 驱动（drivers）距离远，又未单独成「Store」域 |
| `pc_comm` / `rs485_comm` | 调试 / 485 | 与通行、云端平铺，优先级不清 |

结果：想跟「扫码通行」或「云端一条服务」，要在多个平级目录里跳，而不是进一个 `Pass` / `Cloud` 文件夹。

#### 问题 E：根目录噪声大

- 业务分析、IDE 跳转、snprintf 备忘、产品需求 docx 与源码顶层并列。
- 打开工程第一眼不是「分层」，而是「一堆 md / doc」。
- `docs/` 已有部分专文，但根目录与 `USER/` 内仍有流程图、G4G 说明，索引分散。

#### 问题 F：第三方与库的位置不显眼

- `third_party` 可以，但与江协课无对应物；新人不知道和 `FWLIB` 谁优先改。
- `FWLIB` / `CORE` 命名不如 `Library` / `Start` 好记。

#### 问题 G：说明文档与真实结构双轨

- `PROJECT_STRUCTURE.md`、入门规划、`FUNCTION_SUMMARY.md` 中部分仍提旧路径（如 `HARDWARE/`）。
- 另有「文件清单」类文档用于认知填表，容易被误当成「已经重构完」。
- **认知文档 ≠ 目录已改**：当前磁盘仍是旧树。

### 1.3 问题带来的后果（一句话）

> 分层意图其实已有（驱动 / 板级 / 业务），但**命名不江协、硬件双目录、业务未按领域拆、根目录文档噪声**，导致「想整理项目 / 想对照入门课」时仍然迷路。

## 第二部分：修改方案（已批准，待实施）

### 2.1 原则

1. **江协外层骨架**：`User` / `System` / `Hardware` / `Library` / `Start`。
2. **业务显式拆开**：`App/Pass`、`App/Cloud`、`App/Store`、`App/Link`。
3. **Hardware 轻分子目录**（方案 B）：`Serial` / `Storage` / `Time` / `Board` / `Modem`。
4. **第三方**：`Middlewares/`（原 `third_party`）。
5. **R1～R3 只搬家、不改源文件名**；不改业务逻辑与协议。
6. **不删除旧文档文件**：实施时以搬家/更新索引为主；历史 md 迁入 `docs/` 保留内容。
7. **本文档阶段：只新增说明，不实施搬家。**

### 2.2 目标目录树

```text
tikong/
├── User/                 # 原 USER（入口、中断、Keil）
├── System/               # 原 SYSTEM/delay、sys（R1）；usart 在 R2 迁出
├── Hardware/
│   ├── Serial/           # 原 SYSTEM/usart 或 System/usart
│   ├── Storage/          # iic / eeprom / 24cxx
│   ├── Time/             # RTC / ds1302 / DS3231
│   ├── Board/            # HC595、ext_io、timer、floor_ctrl、led、rly、wdg、malloc…
│   └── Modem/            # 4G 驱动
├── App/
│   ├── Pass/             # qr_comm、cmd、data_handler
│   ├── Cloud/            # g4g_comm、Live_data*、uart3_at_sequence
│   ├── Store/            # blackList
│   └── Link/             # pc_comm、rs485_comm、app_comm.h
├── Middlewares/          # cJSON、sha1
├── Library/              # 原 FWLIB
├── Start/                # 原 CORE + system_stm32f4xx.*
└── docs/                 # 说明与流程图（根目录杂 md 迁入后仍保留文件）
```

### 2.3 现状 → 目标对照（摘要）

| 现状 | 目标 | 解决哪类问题 |
| --- | --- | --- |
| `USER/` | `User/` | A 命名 |
| `SYSTEM/delay`、`sys` | `System/` | A |
| `SYSTEM/usart` | `Hardware/Serial/` | B、C |
| `drivers/` + `board/` | `Hardware/{Storage,Time,Board,Modem}/` | B |
| `app/comm`+`utils` 平铺 | `App/Pass\|Cloud\|Store\|Link` | D |
| `third_party/` | `Middlewares/` | F |
| `FWLIB/` / `CORE/` | `Library/` / `Start/` | A、F |
| 根目录 / USER 内杂 md | `docs/`（**保留文件，只搬家**） | E、G |

完整对照与依赖方向见设计文档 §3～§5。

### 2.4 依赖方向（搬家后仍遵守）

```text
User → App / Hardware / System
App  → Hardware / Middlewares / System（串口缓冲头）
Hardware → System / Library
System → Library
Hardware/Serial  禁止依赖 App
```

### 2.5 实施路径（二选一，均未强制执行）

| 路径 | 文档 | 说明 |
| --- | --- | --- |
| **推荐：旁路重建** | [从零重建江协风格工程指南.md](./从零重建江协风格工程指南.md) | 新建 `tikong_jx` 等空树 → 按表拷贝 → 重配 Keil；**旧工程整份保留** |
| 备选：原地改名 | [superpowers/plans/2026-07-17-江协风格目录重构.md](./superpowers/plans/2026-07-17-江协风格目录重构.md) | R1→R4 改现有目录与 uvprojx（路径改动多，易漏） |

#### 若选原地改名时的阶段表

| 阶段 | 做什么 | 验收 |
| --- | --- | --- |
| **R1** | 外层改名：`USER→User`、`SYSTEM→System`、`FWLIB→Library`、`CORE→Start`；`system_stm32` 进 `Start`；文档往 `docs/` 归位（不删） | Keil 能编过 |
| **R2** | `System/usart`→`Hardware/Serial`；`drivers`+`board`→`Hardware/...`；`third_party`→`Middlewares` | 能编过；串口行为不变 |
| **R3** | `app`→`App/Pass\|Cloud\|Store\|Link` | 能编过；能指到通行/云端目录 |
| **R4** | 更新 `PROJECT_STRUCTURE.md`、入门规划、清单路径等（旧文保留或加「历史对照」） | 文档路径与磁盘一致 |

原地改名的勾选细节见 plans 文档。

### 2.6 明确不做（本方案范围外）

- 不改函数逻辑、协议、DMA/中断行为。
- R1～R3 不改 `.c/.h` 文件名（不强制 `Delay.c` 等江协驼峰）。
- 不删除 `DS3231` 等可能残留源码（除非另文确认废弃）。
- **当前不实施**：只新增本文与计划/设计，磁盘结构保持原样。
