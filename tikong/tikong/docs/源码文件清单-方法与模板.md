# 源码文件清单 — 方法与模板

更新时间：2026-07-17

> **目的**：不改代码、不挪目录，通过「按层填表」把工程文件职责理清。  
> **成果物**：[源码文件清单.md](./源码文件清单.md)（你填第一版；之后按同一格式维护）  
> **分层约定**：[PROJECT_STRUCTURE.md](../PROJECT_STRUCTURE.md)  
> **入门节奏**：[梯控固件入门规划-基于F103基础.md](./梯控固件入门规划-基于F103基础.md)（阶段 A～E）

---

## 一、原则（先记住）

1. **只读、只记、不改码**：不重命名、不移动 `.c/.h`，Keil 工程也不动。
2. **先分层，再填表**：一张表对应一层；层内按「入口 → 被调用」顺序排。
3. **头文件优先**：职责与关键入口多数能从 `.h` 和函数名看出；实现细节标「待核」即可。
4. **不确定写「待核」**：比猜错好；后续维护时优先补这些格。
5. **一份格式，长期不变**：列名与填法固定，增删文件只加/删行。

---

## 二、推荐填写顺序

按依赖从「入口」到「库」：

```text
USER  →  app/comm  →  app + app/utils  →  board  →  drivers  →  SYSTEM  →  third_party
         （业务轮询）   （领域逻辑）         （板级）   （片外）    （驱动）   （第三方）
```

| 次序 | 目录 | 你在弄清什么 |
| --- | --- | --- |
| 1 | `USER/` | 上电、`board_init`、`app_init`、`app_poll` |
| 2 | `app/comm/` | 五路串口业务入口：`*Process` |
| 3 | `app/`、`app/utils/` | 通行 / 云端 / 黑名单 / 验签 |
| 4 | `board/` | 定时、RTC 抽象、限层、扩展 IO |
| 5 | `drivers/` | I2C、EEPROM、RTC 芯片、4G、继电器 |
| 6 | `SYSTEM/` | delay / sys / 各 USART 驱动与缓冲 |
| 7 | `third_party/` | cJSON、SHA1（少改） |
| — | `CORE/`、`FWLIB/` | **整层一行**即可，不必逐文件 |

`.c` 与对应 `.h` 可合并一行（路径写 `foo.c / foo.h`），或拆成两行；同一工程内保持一致即可。

## 三、表格列定义（统一格式）

| 列名 | 必填？ | 怎么填 |
| --- | --- | --- |
| **路径** | 是 | 相对工程根，如 `app/comm/qr_comm.c` |
| **一层职责** | 是 | 一句话：这个文件干什么（忌抄文件名） |
| **关键入口** | 建议 | 对外主要函数/宏，2～5 个；没有则写 `—` |
| **依赖层** | 建议 | 它主要调用谁，如 `SYSTEM/usart`、`app/utils`、`board` |
| **被谁用** | 建议 | 谁 `#include` / 调用它；入口文件可写 `main` / `app_poll` |
| **阅读优先级** | 是 | 见下节 `P0`～`P3` |
| **备注** | 可选 | 易混点、旧路径、对应 docs、或 `待核` |

### 阅读优先级

| 级别 | 含义 | 典型例子 |
| --- | --- | --- |
| **P0** | 接手必读 | `main.c`、`app_run.c`、`qr_comm.c`、`cmd.c`、`g4g_comm.c` |
| **P1** | 主业务路径 | `data_handler`、`Live_data_r`、`blackList`、`floor_ctrl` |
| **P2** | 辅助 / 板级 / 驱动细节 | `timer`、`eeprom`、`RTC`、各 `usart*.c` |
| **P3** | 库、兼容残留、暂不用 | `FWLIB`、`CORE`、未接入的 `DS3231` 等 |

## 四、依赖方向（填「依赖层」时对照）

与 [PROJECT_STRUCTURE.md](../PROJECT_STRUCTURE.md) 一致：

```text
app/comm  →  SYSTEM/usart（仅缓冲/标志头文件）、app、drivers、board、third_party
app       →  board、drivers、third_party、SYSTEM（按需）
board     →  drivers、SYSTEM（按需）
drivers   →  SYSTEM（按需）；尽量不依赖 app
SYSTEM/usart  →  不依赖 app（避免循环）
third_party   →  无业务依赖
```

若发现「下层 include 了 app」：备注里写 `依赖方向异常-待核`，先别改代码。

## 五、两条业务线（用来标 P0/P1）

填到 `app/comm` 与 `app` 时，用这两条线自检：

**通行（读头 → 权限）**

```text
UART4/5 DMA+IDLE → RX_Complete
  → QRProcessUart4 / QRProcessUart5（app/comm/qr_comm.c）
  → CommContrl（app/utils/cmd.c）
  → Cmd_Permission_* / 继电器·限层 等
```

**云端（4G → JSON 服务）**

```text
USART3 收包 → G4GProcess（app/comm/g4g_comm.c）
  → parseSerialData（app/Live_data_r.c）
  → 各 parse_* / Reply（Live_data*.c）
```

某文件落在线上 → 至少 **P1**；线上入口 → **P0**。

## 六、建议阅读顺序（对照入门规划 A～E）

| 阶段 | 先填 / 先读哪些行 | 验收感 |
| --- | --- | --- |
| **A** 迁移热身 | `USER/main.c`、`board_init`、`SYSTEM/usart/usart.*`、`delay` | 能编、能打日志 |
| **B** 骨架 | `app_run`、`app/comm/*`、五路 `SYSTEM/usart` | 能说清谁收读头 / 485 / 4G / 调试 |
| **C** 通行 | `qr_comm`、`cmd`、`data_handler`、`rly`/`floor_ctrl` | 能跟到权限命令 |
| **D** 云端 | `g4g_comm`、`Live_data_r`、`Live_data`、`drivers/4G` | 能跟到一条 JSON 服务 |
| **E** 存储联调 | `blackList`、`eeprom`、`board/RTC`、`drivers/ds1302` | 黑名单 / 时间 / 密钥落点清楚 |

细节日程见 [梯控固件入门规划-基于F103基础.md](./梯控固件入门规划-基于F103基础.md)。

## 七、每文件 5 分钟操作法

1. 打开对应 `.h`（无头则看 `.c` 顶部与非 `static` 函数）。
2. 用一句话写 **一层职责**（「做什么」，不是「叫什么」）。
3. 抄 2～5 个对外符号进 **关键入口**。
4. 看 `#include` 填 **依赖层**；全文搜索文件名/关键函数名填 **被谁用**（搜不全就 `待核`）。
5. 对照第五节业务线与第六节阶段，标 **阅读优先级**。
6. 有旧路径/专文/疑惑 → **备注**。

单文件勿超过 10 分钟；卡住就标 `待核` 跳下一行。

## 九、范围约定

| 纳入清单 | 处理方式 |
| --- | --- |
| `USER` / `app` / `board` / `drivers` / `SYSTEM` / `third_party` 业务源码 | **逐文件（或 .c+.h 合并）填表** |
| `CORE` / `FWLIB` | 每层 **一行** 总述 |
| `OBJ` / 编译产物 | **不收录** |
| 根目录 / `USER` / `docs` 下说明 md | 清单文末「文档类」附表，一行一个，不与源码表混排 |

## 十、维护约定（你填完第一版之后）

1. **增文件**：在对应层表格末加一行，列格式不变。
2. **删/弃用文件**：删行，或备注写 `已弃用-日期`。
3. **职责变了**：只改「一层职责 / 关键入口」，并改更新时间。
4. **AI / 协作者补表**：只改 [源码文件清单.md](./源码文件清单.md)；本方法文档仅在列定义或流程变更时改。
5. **与源码冲突时**：以源码为准，改清单并在备注写 `已与源码对齐-日期`。
