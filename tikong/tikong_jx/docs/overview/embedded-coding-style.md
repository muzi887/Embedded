# 嵌入式编程风格（通用入门）

更新时间：2026-07-19

> **定位**：通用 MCU / 裸机 C 习惯说明，帮助建立「能上板、能排障、可维护」的写法。  
> **不是**：某一厂商手册摘要，也不是本仓库命名/中断专文的替代。  
> **本工程对照**：文末 §9；命名见 [naming-conventions.md](./naming-conventions.md)，中断见 [interrupt-service.md](./interrupt-service.md)。

## 1. 运行模型：先想清「谁在跑」

嵌入式程序通常没有「无限线程随便开」的桌面假设。先认清模型：


| 模型 | 特征 | 适合 |
|------|------|------|
| **裸机轮询** | `while(1)` 里依次调业务；中断只收数/打拍 | 中小系统、路径清晰、无 OS 学习成本 |
| **中断驱动** | 事件多在 ISR / 回调里推进 | 强实时边沿；业务仍宜薄 |
| **RTOS** | 多任务、信号量、延时阻塞 API | 多并发、长等待、复杂调度 |

**风格建议**

- 选定一种主模型后**写一致**：不要在轮询工程里到处 `delay_ms(1000)` 堵死所有口。
- 没有明确需求时，**不必为「更现代」上 RTOS**。
- 中断与主循环的分工要固定：**时效在中断，厚度在主循环**（见 §3）。

```text
上电 → 时钟/板级 init → 开中断
         ↓
    while (1) {
        喂狗（若有）
        轮询各通道 / 处理标志位
        非紧急业务
    }
         ↑
    ISR：置标志、拷短缓冲、清硬件标志 —— 尽快返回
```

## 2. 分层习惯：驱动 ≠ 业务

常见三层（名字因工程而异，职责类似）：


| 层 | 职责 | 不该做的事 |
|----|------|------------|
| **板级 / 启动** | 时钟、引脚、外设 init、主循环编排 | 塞进具体产品协议细节 |
| **驱动 / BSP** | 寄存器或库封装：读 DMA、写 GPIO、EEPROM 字节 | 解析业务 JSON、做权限判定 |
| **业务 / App** | 协议、状态机、策略、组帧回执 | 直接散落操作一堆裸寄存器（除非极小 demo） |

**依赖宜单向**：业务 → 驱动 → 库/CMSIS；驱动**不要**回头 `#include` 业务头文件。

好处：换串口芯片时只动驱动；改验签规则时只动业务。

## 3. 中断与主循环

### 3.1 ISR 宜短

**适合在中断里做**

- 读硬件状态、清中断标志  
- 把少量数据写入环形缓冲 / 置 `rx_complete` 一类标志  
- 极短的时间戳或计数  

**尽量不要在中断里做**

- `printf` 大段日志、阻塞 `delay`、等 ACK  
- 复杂 JSON / 加密 / EEPROM 整页擦写  
- 动态分配内存  

主循环看到标志后再做「厚逻辑」。

### 3.2 共享数据

主循环与 ISR 共用的变量：

- 用 **`volatile`**（或工程约定的原子/临界区）标明「可能被异步改」  
- 多字节结构拷贝时，必要时短暂关中断或用「写完再置完成标志」的双缓冲约定  
- 不要假设「读到一半不会被打断」而对长结构无保护地边读边用  

### 3.3 优先级与可重入

- 弄清嵌套：高优先级打断低优先级时，共享资源更要小心。  
- 同一外设不要在 ISR 与主循环里**无约定地双写**寄存器。

## 4. 内存与缓冲

MCU RAM 紧、栈更紧。


| 习惯 | 说明 |
|------|------|
| **大缓冲放静态 / 全局** | 串口 DMA 缓冲、JSON 解析区勿轻易开在深栈函数里 |
| **算清上限** | `buf[N]` 的 N 要对齐协议最大帧；写前检查剩余长度 |
| **慎用 malloc** | 碎片、失败路径、中断里分配都是坑；能静态池就静态池 |
| **字符串当二进制边界处理** | 定长区未必有 `'\0'`；`strncpy` / 自管长度比盲目 `strcpy` 稳 |
| **对齐与 DMA** | 部分芯片要求 DMA 缓冲地址/长度对齐 |

风格上：**每个对外接收缓冲都应能回答「谁写、谁读、写满怎么办、何时清」**。

## 5. 外设访问：寄存器 / 标准库 / HAL

概念级对比（具体 API 因芯片与库版本而异）：


| 风格 | 写法直觉 | 特点 |
|------|----------|------|
| **寄存器** | `GPIOx->ODR \|= …` | 短、快、可读性依赖注释与芯片手册 |
| **StdPeriph 等标准外设库** | `GPIO_SetBits` / `USART_Init` | 函数名贴近外设；国内教材与大量旧工程常用 |
| **HAL / LL** | `HAL_GPIO_WritePin` | 抽象厚、移植路径多；与 StdPeriph **不要混在同一模块乱套** |

**风格建议**

- 一个工程（或至少一个模块）**选定一种**主导风格。  
- 可用 `#define LED_ON() GPIO_ResetBits(...)` 一类 **GPIO 宏** 缩短重复，但宏要简单、带括号、避免有副作用的实参用两次（见 §6）。  
- 初始化集中（时钟 → GPIO → 外设 → NVIC），避免业务函数里「第一次用到才 init」且无文档。

## 6. 宏、位操作与易错点

### 6.1 宏（Macros）

宏是**预处理文本替换**，不是函数：

```c
#define SCLK_H()  GPIO_SetBits(GPIOB, GPIO_Pin_11)
```

**好处**：短名、零调用开销。  
**坏处**：调试看到的是展开后的代码；带参宏若写成 `#define SQR(x) x*x`，则 `SQR(a+1)` 会错——应充分加括号：`((x)*(x))`。

常量宏习惯全大写：`#define MAX_RX_LEN 256`。

### 6.2 volatile

对「硬件寄存器」和「ISR 与主循环共享的标志」使用 `volatile`，否则优化器可能认为「循环里读到的值不会变」而把检查优化掉。

### 6.3 阻塞与看门狗

- 长 `delay`、死等总线 ACK，会拖死轮询与喂狗。  
- 需要等待时：设超时、或改状态机分拍执行。

### 6.4 临界区直觉

「读-改-写」同一寄存器或全局标志时，问自己：中间会不会被中断打断导致丢更新？需要则短关中断或用原子约定。

## 7. 可读性与结构

嵌入式不是「能跑就行」的竞赛；板子上的 bug 往往更难看。


| 习惯 | 说明 |
|------|------|
| **注释写为什么** | 「清 IDLE 标志否则再也进不了中断」比「清标志」有用 |
| **命名见义** | 缓冲、完成标志、超时计数一眼可懂；避免 `a1`/`tmp2` 满天飞 |
| **函数宜短、职责单一** | 一个函数尽量只做「收完一帧」或「根据结果开门」，不要又解析又写 Flash 又发 MQTT |
| **魔法数集中** | 波特率、页大小、协议 case 号用命名常量或集中表格 |
| **错误路径完整** | 解析失败、超时、校验失败要有明确返回/日志，不要静默当成功 |
| **与文件现有风格一致** | 改旧文件时跟其缩进、命名习惯；新文件再贯彻团队规范 |

缩进、大括号风格以团队约定为准；**一致性优于个人偏好**。

## 8. 调试习惯

1. **先有一条可靠日志口**（UART `printf` 或 SWO），开机打印版本/关键 init 结果。  
2. **最小复现**：用固定输入（串口助手一帧、按键一次）复现，再改代码。  
3. **分层排障**：硬件波形 / 是否进中断 → 缓冲是否完整 → 业务分支返回值。  
4. **检查返回值**：I2C/EEPROM/Flash 写失败要可见，不要假设永远成功。  
5. **断点与看门狗**：停太久可能复位；调试期可临时拉长喂狗或关狗（记得改回）。  
6. **改一处验一处**：外设相关改动优先上板，单靠编译通过不够。

## 10. 一页清单（写代码前扫一眼）

- [ ] 清楚当前是轮询还是等中断标志，有没有无超时的死等  
- [ ] 新逻辑放对层（驱动 vs 业务），没有反向依赖  
- [ ] ISR 里没有阻塞与重业务  
- [ ] 大缓冲不在深栈；长度与边界有检查  
- [ ] 共享标志考虑了 `volatile` / 临界区  
- [ ] 失败路径有返回或日志  
- [ ] 风格与周围文件一致；本工程命名/中断规范已扫过链接专文
