# 江协风格目录重构 — 设计说明

日期：2026-07-17  
状态：**已批准；源码侧旁路重建已落地为 `tikong_jx`**（见 [dir-refactor-plan.md](../../setup/dir-refactor-plan.md) 与 plans）  
工程：原设计针对 `tikong/tikong`；现行对照工程为 `tikong/tikong_jx`（STM32F407VET6 / Keil）

---

## 1. 目标与非目标

### 目标

- 外层目录对齐江协入门工程习惯：`User` / `System` / `Hardware` / `Library` / `Start`。
- 业务显式拆开：`App/Pass`、`App/Cloud`、`App/Store`、`App/Link`。
- 目录更清晰，便于对照 F103 江协课接手梯控工程。
- **R1～R3 只搬家、不改源文件名**；逻辑与协议不变。
- 文档集中到 `docs/`；根目录不再堆业务/分析 md。

### 非目标（本设计不做）

- 不改业务逻辑、协议、DMA/中断行为。
- 不强制统一为江协驼峰文件名（如 `delay.c` → `Delay.c` 留后续）。
- 不删除可能仍在用的源码（如 `DS3231.*`）；可标注或移入 `_unused` 子目录仅当后续确认。
- 不重写 Keil 之外的构建系统。

## 2. 方案选择（已确认）

| 项 | 选择 |
| --- | --- |
| 外层风格 | 江协四层 + `App` + `Middlewares` |
| Hardware 内部 | **方案 B**：轻分子目录（`Serial` / `Storage` / `Time` / `Board` / `Modem`） |
| 业务拆分 | `App/Pass`、`App/Cloud`、`App/Store`、**保留 `App/Link`** |
| 文件名 | R1～R3 **只搬家不改名** |
| 实施节奏 | **R1 → R2 → R3 → R4**，每阶段可编译验收 |

## 5. 依赖方向（不变的约定）

```text
User        → App, Hardware, System
App         → Hardware, Middlewares, System（经 Serial 头文件缓冲）
Hardware    → System, Library（按需）
System      → Library
Middlewares → 无 App / 业务依赖
Library / Start → 不依赖 App
```

硬约束：`Hardware/Serial`（原 usart 驱动）**不得**依赖 `App`。

业务线（搬家后路径变，调用关系不变）：

```text
通行：Hardware/Serial(UART4/5) → App/Pass(qr_comm → cmd → data_handler) → Hardware/Board|…
云端：Hardware/Serial(USART3) → App/Cloud(g4g_comm → Live_data_r → …) → Hardware/Modem
```

## 6. 分阶段实施（R1～R4）

### R1 — 外层对齐江协名 + 文档归位

- `USER` → `User`（除 `system_stm32f4xx.*`）
- `SYSTEM` 中 delay/sys → `System/`；usart **暂留**原处或在 R1 末尾再动（推荐 R1 只动 delay/sys + 顶层改名，usart 放 R2）
- `FWLIB` → `Library`
- `CORE` → `Start`；`system_stm32f4xx.*` → `Start`
- 根目录 / `USER` 下说明性 md、流程图 → `docs/`
- 更新 Keil：路径、Include Path、分组名（可先改路径，分组显示名可逐步改）

**验收**：Keil 全编译通过；能烧录；USART1 日志仍可用。

**推荐 R1 边界（降低风险）**：

1. 顶层：`USER→User`、`FWLIB→Library`、`CORE→Start`（含 system_stm32）。
2. `SYSTEM/delay` + `SYSTEM/sys` → `System/`；`SYSTEM/usart` **仍叫 SYSTEM/usart 到 R2**，或整棵 `SYSTEM` 先改名为 `System` 且 usart 仍在 `System/usart` 再于 R2 迁入 Hardware。  
   **本设计采用**：R1 将整个 `SYSTEM` 重命名为 `System`（内部仍含 `usart/`）；R2 再把 `System/usart` → `Hardware/Serial`。

### R2 — Hardware 与 Middlewares

- 建 `Hardware/{Serial,Storage,Time,Board,Modem}`
- `System/usart` → `Hardware/Serial`
- `drivers` + `board` 按对照表迁入
- `third_party` → `Middlewares`
- 删除空目录；更新所有受影响 `#include` 与 Keil Include Path / 文件路径

**验收**：全编译通过；串口收发与初始化行为与改前一致（抽测 UART4 IDLE + 调试口）。

### R3 — App 业务拆分

- 建 `App/{Pass,Cloud,Store,Link}`
- 按对照表迁入；去掉旧 `app/`
- 更新 `#include`、`app_run` 等引用、Keil 分组（建议分组名：`App_Pass`、`App_Cloud`、`App_Store`、`App_Link`）

**验收**：全编译通过；口头能指到：通行在 `App/Pass`，云端在 `App/Cloud`。

### R4 — 文档对齐

- 重写 / 更新：`PROJECT_STRUCTURE.md`
- 更新：`docs/文件说明.md`、入门规划、清单方法中的旧路径
- `源码文件清单.md`：路径列改为新树（内容可仍由接手者填）
- 旧路径对照表保留一小节，避免文档外链失效

**验收**：文档内主要路径可点到真实文件；无大面积 `HARDWARE/`、`app/comm` 误导（除「历史对照」节）。

## 7. Keil 工程注意点

- 工程文件：`User/Template.uvprojx`（随 `USER→User` 移动）。
- Include Path 需覆盖：`User`、`System`（及子路径）、`Hardware/Serial|Storage|Time|Board|Modem`、`App/Pass|Cloud|Store|Link`、`Middlewares`、`Library/inc`、`Start` 等。
- 每个搬家的 `.c` 必须在对应 Group 里更新路径（或移除后按新路径加回）。
- 若使用 `compile_commands.json` / Cursor 索引脚本：R4 或每阶段末跑一次生成脚本。

## 8. 风险与回退

| 风险 | 缓解 |
| --- | --- |
| Include 漏改导致编译失败 | 每阶段只做一类搬家；编译不过不进入下一阶段 |
| uvprojx 相对路径错误 | 改完用 Keil 打开确认；必要时 diff uvprojx |
| 文档与源码双轨 | R4 专门对齐；清单方法文首加「以源码树为准」 |
| 一次改太多无法 bisect | 严格 R1→R4；可用 git commit 每阶段一次（仅当用户要求提交时） |

回退：每阶段开始前保证可编译基线；该阶段失败则还原该阶段文件移动与工程改动。
