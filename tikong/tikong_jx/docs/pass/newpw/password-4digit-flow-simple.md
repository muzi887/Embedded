# 四位访客通行码整个流程

更新时间：2026-07-19

> **本文做什么**：用大白话把新算法 HH-TOTP-4 从头走到尾讲清楚。  
> **正式规格（公式/伪代码）**：[password-4digit-design.md](./password-4digit-design.md)  
> **旧方案问题背景**：见上级目录 [password-4digit-unacceptable.md](../password-4digit-unacceptable.md)

---

## 1. 先记住三句话

1. **同一串 4 位数字**，依次过：园区门 → 单元门 → 电梯。
2. 这 4 位**不是**「千位=园区、末两位=楼层」那种拼盘；整串都是校验码，**看不出楼层**。
3. 钥匙分两层：**园区总钥 `K_c`** → 派生出 **单元钥 `K_u`**；真正算票、验票用的是 `K_u`。

---

## 2. 钥匙：`K_c` 和 `K_u`

```text
园区总钥 K_c（一把，只有本园）
        │
        │  按单元号派生
        ▼
   ┌────┴────┐
   K_u(3单元)   K_u(5单元)  …
   给：3单元门    给：5单元门
       3单元电梯      5单元电梯
```


| 谁            | 手里有什么                           |
| ------------ | ------------------------------- |
| 云 / App      | 能拿到 `K_c`，也能算出任意 `K_u`          |
| 园区大门         | 只有本园 `K_c`（访客选单元后，现场算出对应 `K_u`） |
| 某单元门 / 该单元电梯 | 只有本单元的 `K_u`，**没有**总钥 `K_c`     |


类比：`K_c` 像物业总钥匙模；`K_u` 像某栋单元的专用钥匙。票是用单元钥匙盖的章。

---

## 3. 时间：密码会过期

- 时间切成一段一段，推荐 **每 10 分钟一格**（时间槽）。  
- 出票、验票都按「第几格」来算，不是精确到秒。  
- 设备会试 **当前格、前一格、后一格**（钟差一点点也能过）。  
- 大概 **20～30 分钟** 后旧码失效，要重新生成。

---

## 4. 举例设定

访客要去：

- **A 园区**
- **3 单元**
- **5 楼**
- 现在这一时间槽

App 算出的 4 位码假设是：`**3847`**（实际每次会变，这里只是举例）。

---

## 5. 全程流程

### 步骤① App / 云平台出票

```text
1. 用 K_c 派生出 K_u(3单元)
2. 把这些信息绑在一起算哈希：
     园区A + 单元3 + 楼层5 + 当前时间槽
3. 用 K_u(3) 做 HMAC，截成 4 位 → 得到 3847
4. 发给访客（短信 / App 展示）
```

访客只看到 `3847`，**不知道也不需要知道**里面绑了 5 楼。

---

### 步骤② 园区大门

园区门只知道「我是 A 园」，**一开始不知道**你要去 3 单元还是 5 单元。

**推荐操作（重要）：**

```text
1. 访客先选/扫「3 单元」（屏幕点选，或扫楼栋/单元码）
2. 再输入 3847
3. 门机用 K_c 算出 K_u(3)
4. 用 K_u(3) 试：各个楼层 × 前后时间槽，看能不能算出 3847
5. 能对上 → 开门进园区
```


| 情况                | 结果        |
| ----------------- | --------- |
| 本园、3 单元的真票        | 过         |
| B 园的票拿到 A 园       | 不过（总钥不同）  |
| 5 单元的票，却选成 3 单元来验 | 不过（单元钥不对） |


**为什么要先选单元？**  
若不选，门机只能对「所有单元 × 所有楼层」瞎试，4 位密码太短，**误开门风险很大**。所以产品上要求：园门先确定单元，再验 4 位。

---

### 步骤③ 3 单元门

门机已经预置好 `**K_u(3)`**，访客直接输入 `3847`（不用再选单元）。

```text
1. 用 K_u(3) 试：1楼、2楼…最高层，再加前后时间槽
2. 某一层某一时间能算出 3847 → 开门进单元门厅
```

这里在证明什么？

- ✅ 是 **本单元**、**这个时间附近** 的合法访客  
- ❌ **不**在单元门强制「你只能去 5 楼」

原因：人要先进入单元，再乘电梯；**锁楼层是电梯的事**。


| 情况             | 结果                            |
| -------------- | ----------------------------- |
| 3 单元的票         | 过                             |
| 5 单元的票拿到 3 单元门 | 不过（钥匙是 `K_u(5)`，门只有 `K_u(3)`） |


---

### 步骤④ 电梯

电梯同样持有 `**K_u(3)`**，访客再输 `3847`。

```text
1. 同样试各个楼层 × 前后时间槽
2. 找出「哪一层」能算出 3847
3. 如果恰好只有一层对上（应是 5 楼）→ 只呼这一层
4. 对不上，或碰巧多层都对上 → 拒绝（防歧义）
```


| 情况            | 结果                    |
| ------------- | --------------------- |
| 合法票 `3847`    | 呼 **5** 楼             |
| 乱改成别的 4 位     | 一般呼不了                 |
| 旧方案那种「只改楼层数字」 | **新方案做不到**：码里没有明文楼层可改 |


**真正锁楼层的，是这一步。**

## 6. 一张总图

```text
                    App 出票
         K_c → K_u(3) → 算出 3847
         （绑：园A + 单元3 + 层5 + 时间）
                         │
                         ▼  同一串 3847
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
     园区大门          3单元门           电梯
  先选「3单元」        已有 K_u(3)      已有 K_u(3)
  K_c→K_u(3) 验票       验票开门         验票并找出楼层
  放进园区              放进门厅          只呼 5 楼
```

## 7. 和旧方案差在哪


|         | 旧方案                     | 新方案                |
| ------- | ----------------------- | ------------------ |
| 4 位长什么样 | 分段拼：有的位是校验，有的位是**明文楼层** | 整串校验码，**看不出楼层**    |
| 电梯      | 单元段对了就过，**按密码里的楼层去呼**   | 整票验算，**算出唯一授权层再呼** |
| 改楼层数字   | 单元段仍对 → **可能呼错层**       | 没有楼层字段可改；乱改≈废票     |
| 跨单元     | 往往只比一小段                 | 不同单元不同 `K_u`，整票对不上 |
| 园区门     | 通常只比更少的位                | 建议先选单元，再验整票        |


---

## 8. 落地时必须记住的三条

1. **园区门**：先选/扫单元，再输 4 位（否则误开风险大）。
2. **单元门**：只保证「本单元合法访客」；**锁层靠电梯**。
3. **输错次数限制**：4 位只有 10000 种，不限制就会被盲猜；例如 5 次失败锁定一段时间。
