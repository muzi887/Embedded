# 四位口令：不可接受问题的解决思路

更新时间：2026-07-19

> **问题清单**：[password-4digit-unacceptable.md](./password-4digit-unacceptable.md)  
> **现行算法**：[password-4digit-auth.md](./password-4digit-auth.md)  
> **性质**：对策说明。分 **不改算法（配置/运维/平台约束）** 与 **改算法（固件+平台同步）**；未宣称已在本仓库落地。

---

## 0. 怎么读

[password-4digit-unacceptable.md](./password-4digit-unacceptable.md) 里四类事故，对策一一对应：


| 问题 | 本文章节 | 推荐优先级 |
| --- | --- | --- |
| ① 改楼层位越权呼梯 | §1 | 先：`CallOne` 前查限层；彻底：楼层绑摘要 |
| ② 多设备 code/公钥相同互开 | §2 | **立刻做**：配置唯一性（无需改算法） |
| ③ 一码跨类型误开 | §3 | 密钥隔离 + 业务隔离；彻底则改编码 |
| ④ 无法按人作废 | §4 | 要按人撤销就必须改源串/黑名单 |

原则：能配置解决的不先动四位编码；动算法必须 **App/平台与固件同一天切换**，否则全场口令失效。

---

## 1. 对应①：阻止「改楼层位仍能呼梯」

### 目标

单元位通过后，不能只凭用户改 `pwd%40` 就呼到任意楼。

用户「随意改」的含义：口令只是 4 个数字（如 `2285`），改成 `2292` 再刷读头；固件仍可能单元位校验通过。  
**口令算法拦不住这种改法**，只能靠「呼梯前再卡一道」或「把楼层绑进摘要」。

### 方案 A — 不改四位编码，呼梯前用限层卡住（推荐先做）

现状（重要）：口令成功后的路径是

```text
单元位通过
  → Floor_AuthCheck_limit_off(限层)
  → Floor_CallOne(pwd % 40)    ← 当前直接打继电器，未判断该层是否在限层位图内
  → Floor_AuthCheck_limit(限层)
```

因此：**只「配严限层」、不改这一处调用，并不能保证改成 12 楼就呼不成。**  
限层位图今天主要管「哪些层常开授权」，**没有**在 `Floor_CallOne` 前做允许列表检查。

要真正挡住「乱改楼层位」，固件加一小步（仍不改口令生成规则）：

```text
floor = pwd % 40
若 floor 不在 FloorCtrl 限层位图内 → 鉴权失败 / 不 CallOne，beep×2
否则 → Floor_CallOne(floor)
```

| 动作 | 说明 |
| --- | --- |
| 固件 | `qr_handle_password_input` 电梯分支：`CallOne` 前查限层位图 |
| 配置 | 按户/按梯把限层配成甲只含 5 楼等；平台发码楼层 ⊆ 限层 |
| 验收 | 合法 `2285` 能呼 5；改成 `2292` 被拒绝、不打 12 楼继电器 |

**效果**：他仍可改数字，但**改到限层外的楼会失败**。依赖限层配得对；限层配成「全开」则仍拦不住。

### 方案 B — 改算法：楼层进入校验

思路：呼梯楼层必须与摘要绑定，改楼层位 → **单元都过不了**（或整码作废）。

示例（需平台同步）：

```text
源串增加楼层，或比对增加楼层约束，例如：
  auth_src = code,publicKey,时间,floor
  或：在通过单元位后，再要求 floor == f(sum)   // 由平台与固件约定同一 f
```

| 改动面 | 内容 |
| --- | --- |
| 固件 | `qr_handle_password_input` 电梯分支：除 `sum%25` 外校验楼层；或源串拼入楼层后再摘要 |
| 平台 | 生成四位时用同一规则；用户改楼层位则整码作废 |
| 代价 | 一户多楼层要发多码，或加长口令/换编码 |

**效果**：从根上消除「只改楼层仍过」；工作量大，属协议升级。

## 2. 对应②：阻止「多设备材料相同而互开」

### 目标

不同物理门/梯的口令空间隔离。

### 方案 A — 不改算法（必须做）

算法已假定 `g_device_code` + `g_device_public_Key` 唯一标识设备：

```text
auth_src = code,publicKey,时间
```

| 动作 | 说明 |
| --- | --- |
| 出厂/配网一机一码、一机一钥 | EEPROM `PUBLIC_DEVICE_CODE` / `PUBLIC_KEY` 不复用 |
| 平台建档校验唯一 | 禁止两台设备登记相同 code+公钥 |
| 现场巡检 | 抽两台机打印 `auth source str` 前缀，确认 code/公钥不同 |

**效果**：材料不同 → `sum` 不同 → 互开消失。**不改固件即可闭环**，应作为默认防线。

### 方案 B — 加固（可选）

源串再拼入不可复用的设备 ID（若与 code 分离），例如：

```text
auth_src = deviceId,code,publicKey,时间
```

仅当「code 可能重复、但 deviceId 唯一」时有意义；若 code 已唯一，收益有限。

## 3. 对应③：阻止「电梯码误开园区门」等跨类型

### 目标

发给电梯的口令，不能在园门/楼门上仅因某一段碰巧相等就通过。

### 方案 A — 不改四位宽度（组合拳）

| 动作 | 说明 |
| --- | --- |
| 不同类型设备独立 code/公钥 | 降低「同源 sum」；模值仍可能碰巧撞，但概率下降 |
| 平台按用途分发、UI 标明 | 用户不易拿电梯码去刷园门 |
| 读头/场景隔离 | 园门读头不展示电梯口令入口（产品层） |

**效果**：降低误用；**不能从数学上消灭**「园门 `sum%10` 碰巧等于电梯码千位」。

### 方案 B — 改算法：类型进入源串或分码

**B1. 源串带设备类型（平台按目标 type 生成）：**

```text
auth_src = code,publicKey,时间,type
```

- 园门固件用 `type=1` 算 sum；电梯口令用 `type=3` 生成  
- 把电梯码拿到园门：园门按 type=1 算的 sum 与码内段对不上（在材料独立时）  

注意：若用户改的是「同一材料下不同 type 的模值碰巧撞段」，仍要靠材料隔离；B1 主要拆开「同材料不同 type 共用一个 sum」的问题。

**B2. 分通道口令（更干净）：**

- 园门只发「院位」短码或独立编码；电梯另发含单元+楼层的码  
- 或口令增加用途前缀（需改读头 JSON/`data` 长度约定）

| 改动面 | 内容 |
| --- | --- |
| 固件 | 拼源串加 `g_device_type`；或分类型解析不同口令格式 |
| 平台 | 按目标设备 type 生成；禁止「一码打天下」 |
| 代价 | 用户可能要持多码；协议与 App 同步升级 |

## 4. 对应④：支持「作废某一个用户」

### 目标

甲离职后，甲的通行立即失效，乙、丙同槽不受影响。

### 方案 A — 不改四位摘要（旁路能力）

现行口令算法**做不到按人撤销**。可叠加其它通行能力：

| 动作 | 说明 |
| --- | --- |
| 主通行改走权限二维码/`CommControl` | 带用户/卡号，可走黑名单（见通行权限链文档） |
| 口令仅作访客短时码 | 接受「同槽共享」，不作废到人 |
| 紧急：换该设备公钥 | 全员口令失效后重发——伤乙丙，仅应急 |

### 方案 B — 改算法：源串或口令含用户维

```text
auth_src = code,publicKey,时间,userId
```

或口令加长，嵌入用户相关校验段。

| 改动面 | 内容 |
| --- | --- |
| 固件 | 拼串含 userId；或维护用户口令黑名单表（EEPROM/云同步） |
| 平台 | 按人生成；作废 = 黑名单 / 停发该 userId |
| 代价 | 四位空间更紧或要加长；黑名单要有存储与下发 |

**效果**：才能真正「只作废甲」。属产品能力升级，不是配配置能补上的。

## 5. 落地顺序建议


| 顺序 | 做什么 | 对应问题 | 改固件？ |
| --- | --- | --- | --- |
| 1 | 一机一 code、一机一公钥 + 平台唯一性校验 | ② | 否 |
| 2 | 电梯限层按户配严 + 改楼层位验收 | ① | 否 |
| 3 | 类型间密钥隔离 + 产品勿一码多用 | ③ | 否 |
| 4 | 明确口令不做「按人撤销」；要则上权限码/黑名单 | ④ | 视方案 |
| 5 | 若产品强要求：楼层绑摘要 / type 进源串 / userId | ①③④ | **是** |

---

## 6. 改算法时的共同约束

无论采用哪条「方案 B」：

1. **平台与固件同一规则、同一天切换**，否则旧口令全军覆没或双向不通  
2. 改 `auth_src` 或比对式后，更新 [password-4digit-auth.md](./password-4digit-auth.md) 与碰撞/不可接受文档  
3. 三时刻容差（`drift`）逻辑可保留，但源串字段变更后要重做联调日志对照  
4. `Middlewares/sha1` 可不动；改的是 `qr_build_password_auth_digest` / `qr_handle_password_input` 与平台生成侧
