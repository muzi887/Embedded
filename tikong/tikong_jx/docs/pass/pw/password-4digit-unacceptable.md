# 四位口令：梯控场景下「不可接受」的重复

更新时间：2026-07-19

> **算法**：[password-4digit-auth.md](./password-4digit-auth.md)  
> **一般性碰撞原理**：[password-4digit-collision.md](./password-4digit-collision.md)  
> **本文立场**：按梯控现场（园区门 / 楼门禁 / 电梯）判断——哪些「重复」可接受，哪些会造成**越权或错控**。

---

## 1. 先划清：场景里可接受的重复

本系统口令是「同设备 + 同时间槽」的短时通行码，**不是**每人一把全局唯一密钥。因此下面情况**通常可接受**：


| 情况                           | 为何可接受                            |
| ---------------------------- | -------------------------------- |
| 同一用户、同一电梯、同一时间槽，多次打开口令相同     | 源串材料不变，本来就该一样                    |
| 同一园区门、同一时间槽，多个用户千位相同（甚至整码相同） | 园区门只认 `sum%10`；同园同槽本就是同一把「门禁校验位」 |
| 同一楼门禁、同一时间槽，多个住户单元位相同        | 楼门只认 `sum%25`；同设备同槽共用同一鉴权段       |


真正要警惕的，是下面几类——会变成**开错门、呼错层、或跨设备通行**。

---

## 2. 不可接受①：电梯口令被改楼层后仍能呼梯（越权呼层）

### 机制

电梯鉴权**只比对单元位**：

```text
sum % 25 == (pwd % 1000) / 40
```

楼层位 `pwd % 40` **不参与**摘要校验，成功后直接：

```text
Floor_CallOne(pwd % 40)
```

### 具体例子（不可接受）

解码公式（与固件一致）：

```text
院   = pwd / 1000
单元位       = (pwd % 1000) / 40
楼层位       = pwd % 40
```

等价编码：`pwd = 千位*1000 + 单元*40 + 楼层`（单元 0～24，楼层 0～39）。

某槽设备算出单元位应为 `7`。平台发给住户甲的合法口令是：

```text
2285
  院 = 2285 / 1000 = 2
  单元 = (2285 % 1000) / 40 = 285 / 40 = 7
  楼层 = 2285 % 40 = 5
  → 意图：只能去 5 楼
```

甲（或他人）把口令改成（只改楼层，单元仍为 7）：

```text
2292
  院 = 2
  单元 = (2292 % 1000) / 40 = 292 / 40 = 7   ← 仍对
  楼层 = 2292 % 40 = 12                      ← 被改成 12
```

固件：单元位仍对得上 → **鉴权通过** → `Floor_CallOne(12)`。  
只要限层配置未挡住 12 楼，就会呼到非授权楼层。

**这不是「多人密码一样」的无害重复，而是「鉴权段对了就可以改楼层」的越权。**

缓解（产品/配置侧，非当前算法本身）：

- 限层 `FloorCtrl_GetLimit` 必须按户配严  
- 平台勿把「可呼楼层」只写在口令楼层位里却不靠限层兜底  
- 若要「楼层也绑进摘要」，需改算法（当前固件未做）

## 3. 不可接受②：多台设备共用设备码 + 公钥 → 跨门互开

### 机制

口令摘要的源串**只有三样**，没有「机身序列号」或其它按台区分的字段：

```text
auth_src = "{g_device_code},{g_device_public_Key},{YYYYMMDDhhmm}"
sum      = SHA1(auth_src) → 左10 ASCII 累加
```

固件比对时用的是本机 RAM 里的 `g_device_code` / `g_device_public_Key`（来自本机 EEPROM）。因此：

```text
若设备 A、B 存了相同的 code + 公钥（平台存储错误）
且墙钟落在同一时间槽（同一 YYYYMMDDhhmm）
    → A、B 算出的 auth_src 字符串完全相同
    → sum 相同
    → sum%10 / sum%25 相同
    → 同一串四位口令在 A、B 上鉴权段都合法
```

算法**假定**「code+公钥」全局唯一标识一台逻辑设备；材料一复用，口令空间就合并，固件无法分辨「这串是发给 A 的还是 B 的」。

### 具体例子（不可接受）

- A 栋电梯与 B 栋电梯误配成同一 `device_code` / 公钥  
- 住户拿到按 A 栋材料生成的口令（例如单元位对上 `sum%25`），到 B 栋刷 → B 用相同材料算出相同 `sum` → **单元位同样通过** → 可能开门/呼梯  

**同用户同电梯同码可接受；不同楼栋因配置撞车而互开，不可接受。**

缓解：每台设备独立 `g_device_code` 与 `g_device_public_Key`，出厂与配网时校验不重复。

## 4. 不可接受③：同一串口令跨设备类型误开（拆位语义冲突）

### 机制

平台把三种语义**打进同一个四位数**，但固件按 `g_device_type` **只验其中一段**，且摘要里**没有「口令用途/设备类型」字段**：

```text
pwd = 院×1000 + 单元×40 + 楼层

type "1" 园区门：只判  院 == sum%10
type "2" 楼门  ：只判  单元 == sum%25
type "3" 电梯  ：只判  单元 == sum%25，然后用楼层呼梯

源串仍是：code,publicKey,时间   ← 不含 type，也不含「这串只许开电梯」
```

因此固件看到的只是「四个数字 + 本机 type + 本机算出的 sum」，**不会**判断「用户本意是刷电梯还是刷园门」。  
只要本机所验的那一段碰巧对上，就会授权。

跨类型还能碰上「材料独立但模值撞车」：园门的 `sum_门%10` 与电梯口令的千位（院）数值相同即可误开——不必 code/公钥相同（相同则更严重）。

### 具体例子（不可接受）

电梯口令（单元 7、楼层 5，院位 2）：

```text
2285 → 院=2，单元=7，楼层=5
```

用户拿到附近一台**园区门**上刷。园区门只做：

```text
sum_门 % 10 == 2285/1000 == 2 ？
```

若该门当前槽（或邻槽）算出的 `sum_门 % 10` 恰好是 `2`，则**开门成功**。  
固件不会问「2285 是不是电梯码」。

若园门与电梯还共用 code/公钥，则两边 `sum` 同源，跨类型误开更容易出现。

**多人刷同一个园区门可接受；「电梯码误开园区门」是拆位 + 分类型只验一段导致的边界打穿，不可接受。**

缓解：读头/业务区分用途；不同类型设备独立密钥；长期避免「一码多用、各设备只验一段」。

## 5. 不可接受④：需要「作废某一个用户」时无法单独失效

### 机制

合法与否完全由下面决定，**全程没有用户 ID**：

```text
源串 = code + 公钥 + 时间槽     ← 设备级，不是用户级
sum  = 对该源串做摘要
比对 = 只看 sum 模运算后的「鉴权段」是否等于口令里对应段
```

因此在**同一台设备、同一个时间槽**内：

```text
所有用户共享同一份 auth_src
    → 同一个 sum
    → 同一个合法鉴权段（园门：同一个千位；楼/梯：同一个单元位）
```

固件口令路径里没有「甲的口令 / 乙的口令」之分，只有「这段数字此刻对不对」。  
作废甲 = 必须让这段鉴权失效 = 改公钥 / 等槽过期 / 关设备 → **乙、丙同槽一并失效**。

这与第 1 节「多人同码可接受」不矛盾：通行可以共享校验位；**按人撤销**需要源串或比对里出现用户维，当前算法没有。

### 具体例子（运维上不可接受）

住户甲离职，希望立刻作废甲的 `2285`，但乙、丙同设备同槽的合法口令鉴权段与甲相同（例如单元位都是 7）。  
设备无法识别「刷的是甲还是乙」。只能换公钥或等过期——乙、丙也会受影响。

---
## 6. 对照小结


| 现象                 | 本项目是否可接受               |
| ------------------ | ---------------------- |
| 同用户、同电梯、同时槽，密码一样   | 可接受                    |
| 同园区门、同时槽，多用户千位一样   | 可接受                    |
| 改电梯口令楼层位后仍通过并呼到别的层 | **不可接受（越权）**           |
| 两台设备 code/公钥相同导致互开 | **不可接受（跨设备）**          |
| 电梯/楼门口令在园区门上误开     | **不可接受（跨类型）**          |
| 要单独拉黑某一用户的口令       | **当前做不到（运维不可接受则需改方案）** |
