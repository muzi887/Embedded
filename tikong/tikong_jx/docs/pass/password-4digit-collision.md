# 四位口令：重复 / 碰撞问题说明

更新时间：2026-07-19

> **算法全文**：[password-4digit-auth.md](./password-4digit-auth.md)  
> **源码**：`App/Pass/qr_comm.c` → `qr_handle_password_input`  
> **性质**：已知局限说明（非使用手册）。平台端逻辑较简单时，源串与口令更容易出现「看起来该不同、实际却相同」的情况。

---

## 1. 问题是什么

现场或联调时可能观察到：

- **不同时段、不同人**，刷出的四位口令**相同**  
- 平台按规则新算一串，结果与上一槽或另一设备**撞车**  
- 固件侧「验过了」，并不代表口令在全局是唯一的  

本文把「重复」拆成两类原因：**平台源串重复**，以及**固件算法空间本身很小**（即使源串不同，比对用的那一位也可能相同）。

---

## 2. 平台侧：源串可能重复

固件摘要源串形态固定为：

```text
{g_device_code},{g_device_public_Key},{YYYYMMDDhhmm}
```

平台若逻辑较简单，容易在以下情况生成**相同源串**（或等价到同一时间槽后相同）：

| 情况 | 结果 |
| --- | --- |
| 同一设备、同一时间槽内多次下发 | 源串完全相同 → `sum` 相同 → 四位编码相同 |
| 时间只粗到「槽」、未区分用户/会话 | 多人在同一槽共用同一套材料 → 口令相同 |
| 设备码 / 公钥未按现场区分，或多处复用同一套 | 不同门禁也可能算出相同源串 |
| 时间槽对齐规则与固件不一致，却碰巧落在同一代表时刻 | 源串仍可能撞车 |

源串一重复，后面的 SHA1 与 `sum` 必然重复——这不是固件算错，而是**输入材料相同**。

---

## 3. 固件侧：比对空间很小（算法固有碰撞）

即便源串不同，固件也**不是**拿完整 SHA1 或完整 `sum` 去和四位数字全体比对，而只比「拆出来的一小段」：

| 设备类型 | 实际比对式 | 有效取值个数 |
| --- | --- | --- |
| 园区门 `"1"` | `sum % 10` == 千位 | **仅 10 种**（0～9） |
| 楼门禁 `"2"` / 电梯 `"3"` | `sum % 25` == 单元位 | **仅 25 种**（0～24） |

因此：

```text
很多不同的源串 / 不同的 sum
        │
        ▼
   只保留 sum % 10 或 sum % 25
        │
        ▼
   大量输入映射到同一个比对值  →  口令「重复」或「误认为同一把」
```

补充点：

1. **四位数里很多位不参与鉴权**  
   - 园区门只看千位；后三位只要凑成合法数字即可，固件不验其与摘要的关系。  
   - 楼门禁/电梯只看单元位 `(pwd%1000)/40`；电梯的 `pwd%40` 只用于呼梯，**不参与** `sum` 比对。  
   → 不同的完整四位数，鉴权段可以相同。

2. **三时刻容差放大「可过」集合**  
   当前槽 / 前一槽 / 后一槽 任一 `S` 命中即通过。同一四位数可能对应多个可接受的时间槽，跨槽时也更容易出现「还是那几个数字」。

3. **SHA1 本身几乎不撞，模运算会撞**  
   40 位十六进制哈希极少碰撞；碰撞主要发生在 **ASCII 累加后再 `%10` / `%25`** 这一步。

---

## 4. 直观例子

**园区门**：某槽算出 `sum % 10 == 3`。  
则千位为 `3` 的口令（`3xxx`，共约一千种写法）在鉴权意义上都「对」——固件只认千位是不是 3。

**楼门禁**：某槽算出 `sum % 25 == 7`。  
则单元位为 7 的所有四位数（`pwd` 满足 `(pwd%1000)/40 == 7`）鉴权都通过；同一单元位可对应多种楼层位编码。

**平台源串重复**：同一设备在 14:30 槽内两次生成，源串都是 `CODE,KEY,202607191430` → 两次四位必然相同。

---

## 5. 和「安全强度」的关系（接手预期）

本算法定位接近：**短时、同设备、可容忍时钟偏差的通行校验码**，不是高熵、全局唯一的密钥。

| 预期 | 是否成立 |
| --- | --- |
| 同设备、同时槽，口令可复现 | 是（设计如此） |
| 全局不重复、不可猜测 | **否**（模 10/25 + 平台源串简单） |
| 防止任意外人长期撞库式试探 | 弱（尤其园区门仅 10 态；另有读头与蜂鸣等现场因素） |

若产品要求「每用户每分钟唯一」「不可跨用户复用」，需要在**平台源串**中增加区分字段（用户 ID、随机 nonce、权限序号等），并同步改固件摘要拼串与比对位宽——当前固件未做这些。

---

## 6. 联调时如何判断「是哪一种重复」

| 观察 | 更可能的原因 |
| --- | --- |
| USART1 里 `auth source str` 两次完全一样 | 平台源串重复（或同槽重入） |
| 源串不同，但 `ascii sum % 10`（或 `% 25`）相同 | 算法模运算碰撞 |
| 源串不同、模值也不同，但口令后几位不同、鉴权段相同 | 未参与比对的位在变，属拆位设计，不是摘要撞车 |

日志关键字见 [password-4digit-auth.md](./password-4digit-auth.md) 第 10 节。
