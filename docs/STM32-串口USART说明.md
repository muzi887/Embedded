## 一、通信、通信协议、通信接口

学串口前先分清三个概念，避免把「线」「规矩」「整次传数」混在一起。

**通信（Communication）**：两个或两个以上实体之间，按照约定方式进行信息交换与传输的过程。例：MCU 把温度测量结果发给电脑。

**通信协议（Communication Protocol）**：为保证通信双方能正确理解对方所传内容，而预先约定的一套规则集合，通常包括数据格式、时序、检错与应答等。例：波特率 9600 与帧格式 8N1、Modbus、自定义帧 `@…\r\n`。

**通信接口（Communication Interface）**：实现通信所依赖的物理与电气连接规范，规定信号电平、线制、引脚定义及连接器形式等。例：TTL 串口、RS-232、RS-485、USB。**通信端口 Communications Port**。

三者关系可理解为：通信是过程，协议是规则，接口是承载规则的物理/电气手段。

入门调试里常见叠法：接口用 MCU ↔ USB-TTL（**TTL 电平**）；链路习惯用 8N1、9600/115200；应用层用 `printf` 文本或 `@LED_ON\r\n` 一类自定义帧。

## 二、串口与电平：TTL / RS-232 / RS-485

### 2.1 串口是什么

**串口（Serial Port）**：一种串行通信接口，数据按比特顺序在传输线上**逐位传送**（相对并行口「多位同时」）。日常所说串口多指 **异步串行**（UART/USART 风格）：使用 TX、RX 信号线，双方约定波特率，无共用时钟线；接线时本机 TX 接对方 RX，本机 RX 接对方 TX，并通常共地。

常见通信方式对照：


| 名称        | 引脚                | 双工  | 时钟  | 电平  | 设备  |
| --------- | ----------------- | --- | --- | --- | --- |
| **USART** | TX、RX             | 全双工 | 异步  | 单端  | 点对点 |
| **I2C**   | SCL、SDA           | 半双工 | 同步  | 单端  | 多设备 |
| **SPI**   | SCLK、MOSI、MISO、CS | 全双工 | 同步  | 单端  | 多设备 |
| **CAN**   | CAN_H、CAN_L       | 半双工 | 异步  | 差分  | 多设备 |
| **USB**   | DP、DM             | 半双工 | 异步  | 差分  | 点对点 |


说明：表中 USART 按入门最常用的 **异步全双工** 写法；片上 USART 外设另可配置同步模式（见第四章）。

电平标准一致才能进行通信。

### 2.2 三种常见电平

电平标准描述的是 **线上电压怎么表示 0/1**。同一套「起始位 + 数据位」时序，可以跑在不同电平上。


| 标准         | 逻辑习惯（常见）                          | 电压量级              | 线制特点          | 入门场景                 |
| ---------- | --------------------------------- | ----------------- | ------------- | -------------------- |
| **TTL**    | 常：高≈3.3V/5V 为空闲（1），低为 0           | 0 / 3.3V 或 0 / 5V | 单端，短距         | **MCU 直连模块、USB-TTL** |
| **RS-232** | 负电压为逻辑 1，正电压为逻辑 0（与 TTL 极性相反）     | 约 ±3～±15 V        | 单端，PC 九针口时代常见 | 老工控机、需电平转换芯片         |
| **RS-485** | 用 **A/B 差分** 表示位，不是「一根线对 GND 的高低」 | 差分几百 mV 级有效       | 半双工总线，可多站     | 现场 **Modbus**、长线     |


```text
MCU（TTL）──USB-TTL──电脑串口助手     ← 入门最多
MCU（TTL）──MAX232──RS-232 设备
MCU（TTL）──MAX485──A/B 总线──多台从站
```

### 2.3 USB-TTL 与三个「口」

入门调试里，**USB-TTL 模块**（常见芯片 **CH340**、CP2102、FT232 等）做的事是：把 MCU 侧的 **TTL 串口信号**（TX/RX 高低电平）转成电脑能识别的 **USB 虚拟串口**。模块本身不是 STM32 的 USART，而是中间的 **电平 + 协议桥**。

```text
STM32（USART1，TTL）──TX/RX/GND── USB-TTL 模块 ──USB 线── 电脑
                                      │
                               CH340 等桥接芯片
                                      │
                         操作系统枚举为「COMx 虚拟串口」
                                      │
                              串口助手打开 COMx 收发
```

同一块板子上若还有 **ST-Link**，它的 USB 口走的是 **SWD 调试协议**，用于烧录与断点；和 USB-TTL 的「虚拟串口」是 **两条独立链路**（详见 [STM32-JTAG与SWD说明.md](./STM32-JTAG与SWD说明.md)）。

日常容易把下面三个名字当成同一个东西，其实各管链条上不同一层：


| 名称              | 定义                                          | 所在位置                    | 入门里干什么                         |
| --------------- | ------------------------------------------- | ----------------------- | ------------------------------ |
| **USB 口**       | 电脑的 **物理接口**（Type-A / Type-C 等），传的是 USB 协议帧 | 电脑机身、USB-TTL 模块上的 USB 座 | 插线、供电（部分模块）、让 OS 识别设备          |
| **COM 口**（虚拟串口） | 操作系统里的 **逻辑串口**；串口助手下拉框里的 `COM3`、`COM5` 等   | Windows 设备管理器、串口调试工具    | 选端口、设波特率，与助手收发字节               |
| **USART**       | STM32 片上的 **硬件串口外设**，按异步时序在 TX/RX 引脚逐位收发    | MCU 芯片内，引脚如 PA9/PA10    | 固件里 `USART_Init`、`printf`、中断收包 |


三者关系可记为：

> **USB 口** = 电脑这一端的物理插头；**COM 口** = 软件里看到的逻辑串口名；**USART** = MCU 上真正干活的硬件串口。

数据路径是：**USART 发比特 → TTL 线 → USB-TTL 芯片打包成 USB → 电脑映射成 COM → 串口助手读 COM**。任一层没配对（线没交叉、驱动未装、COM 选错、波特率不一致）都会表现为无输出或乱码。

## 三、串口时序

### 3.1 异步一帧长什么样

双方事先约定：**波特率、数据位、校验、停止位**（如 **9600 8N1** = 9600 bit/s，8 数据位，无校验，1 停止位）。

发送一字节时，线上常见顺序：

```text
空闲（线保持高）
    │
    ▼
起始位 START（拉低 1 个比特时间）  ← 告诉对方「开始了」
    │
    ▼
数据位 D0 → D1 → … → D7           ← 低位先发（LSB first）
    │
    ▼
（可选）校验位 Parity
    │
    ▼
停止位 STOP（拉高 1 或 2 个比特时间）
    │
    ▼
空闲（继续为高）或下一帧起始位
```

```text
电平示意（TTL，空闲为高）：

     空闲   START  D0 D1 D2 D3 D4 D5 D6 D7  STOP  空闲
       ──┐     ┌───────────────────────────┐     ┌──
         └─────┘                           └─────┘
           低              数据位（示意）           高
```

手册中的帧格式（字长 8/9 位、停止位可选长度）：

![[attachments/stm32-usart-word-length-9-8.png]]

![[attachments/stm32-usart-stop-bits.png]]


| 参数               | 含义                                      |
| ---------------- | --------------------------------------- |
| **波特率 BaudRate** | 每秒传送的符号数；异步串口里常等价于「每秒比特数」量级             |
| **比特时间**         | `T_bit = 1 / BaudRate`（9600 → 约 104 µs） |
| **起始位**          | 固定拉低，用于同步「这一帧的起点」                       |
| **停止位**          | 拉高，给接收方收尾与下一段空闲                         |


收发双方波特率必须一致，否则会出现乱码、丢字节。

接收端通常以 **16 倍过采样** 找起始位、判定每一位，并做噪声过滤：

![[attachments/stm32-usart-start-bit-detection.png]]

![[attachments/stm32-usart-noise-sampling.png]]

### 3.2 和「同步」的差别（为 USART 铺垫）


|      | 异步（常用 UART）   | 同步               |
| ---- | ------------- | ---------------- |
| 时钟线  | **无**；靠起始位对齐  | 常有 **SCK/CK**    |
| 速率约定 | 事先约好波特率       | 跟随时钟边沿采样         |
| 入门调试 | `printf`、串口助手 | 较少；部分 USART 同步模式 |

## 四、USART：通用同步/异步收发器

### 4.1 定义

**USART**（Universal Synchronous/Asynchronous Receiver/Transmitter，**通用同步/异步收发器**）是 MCU 片内外设：既能按 **异步** 方式工作（最常用），也能在部分配置下做 **同步** 收发。

日常口语「开串口」在 STM32 上就是配置 **USARTx**。


| 名称        | 关系                         |
| --------- | -------------------------- |
| **UART**  | 强调异步收发                     |
| **USART** | STM32 外设名；异步是默认用法，另支持同步等能力 |
| **串口**    | 统称；硬件上常落到某个 USART          |


F103 常见：USART1（常挂 APB2，PA9/PA10）、USART2/3 等。本工程调试串口用 **USART1**。

### 4.2 硬件角色

```text
CPU 写 DR ──► 发送移位器 ──► TX 引脚（一位一位出去）
RX 引脚 ──► 接收移位器 ──► DR ──► CPU 读 / 中断取走
```

![[attachments/stm32-usart-block-diagram.png]]


| 能力       | 说明                  |
| -------- | ------------------- |
| 波特率发生器   | 由外设时钟分频得到比特节拍       |
| TXE / TC | 发送数据寄存器空 / 发送完成     |
| RXNE     | 接收数据寄存器非空（常开中断收字节）  |
| 校验、停止位   | 与 `USART_Init` 参数对应 |


### 4.3 配置流程（SPL）

```text
1 开 USART + GPIO 时钟
2 GPIO：TX → 复用推挽 AF_PP；RX → 上拉输入（或浮空，按教程）
3 USART_Init：波特率、字长、停止位、校验、模式 Tx|Rx
4 （收包常用）USART_ITConfig(RXNE) + NVIC
5 USART_Cmd(ENABLE)
```

本工程 [Serial.c](../project/Hardware/Serial.c)：USART1、**9600 8N1**、PA9/PA10、RXNE 中断。


| 函数                                     | 作用                       |
| -------------------------------------- | ------------------------ |
| `USART_Init`                           | 波特率与帧格式                  |
| `USART_SendData` / `USART_ReceiveData` | 写/读数据寄存器                 |
| `USART_GetFlagStatus`（如 TXE）           | 查询标志                     |
| `USART_ITConfig`                       | **IT = Interrupt**，开某类中断 |
| `USART_Cmd`                            | 使能外设                     |


发送一字节典型写法：写入 DR，等 **TXE**（或 TC）再发下一字节。

### 4.4 接线（USB-TTL）


| USB-TTL | 蓝 pill                   |
| ------- | ------------------------ |
| TXD     | PA10（USART1_RX）          |
| RXD     | PA9（USART1_TX）           |
| GND     | GND                      |
| VCC     | 一般不接（板子另供电）；若供电须 3.3V 规则 |


常见 CH340G 模块原理示意（注意模块上 **3.3V / 5V 供电跳线** 与 STM32 电平一致）：

![[attachments/ch340g-usb-ttl-schematic.png]]

三个「口」的分工见 **§2.3**。接线细节见 [入门接线与线材说明.md](./入门接线与线材说明.md)。ST-Link 烧录与 USB-TTL 看日志是两条链路。

## 五、数据包与状态机收包

硬件 USART 一次只保证 **收/发字节**；「一包命令」要靠软件按协议拼起来。

### 5.1 什么是数据包

**数据包（Packet / Frame）**：按约定格式排好的一串字节，表示一条完整消息。


| 常见字段            | 作用                       |
| --------------- | ------------------------ |
| 包头 Header       | 标记「一包开始」（如 `0xFF`、`'@'`） |
| 长度 Length       | 可变长包用来知道后面还有几字节          |
| 载荷 Payload      | 真正数据或命令                  |
| 校验 Checksum/CRC | 查传错（进阶）                  |
| 包尾 Tail         | 标记结束（如 `0xFE`、`\r\n`）    |


### 5.2 固定包长 vs 可变包长


| 类型       | 特点      | 优点               | 缺点               | 例子                      |
| -------- | ------- | ---------------- | ---------------- | ----------------------- |
| **固定包长** | 每包字节数固定 | 状态机简单：收满 N 字节即结束 | 短命令也占满长度；改协议要改 N | `FF + 4 字节数据 + FE`（共 6） |
| **可变包长** | 长度不固定   | 灵活、省流量           | 要用包头/包尾或长度字段界定   | `@LED_ON\r\n`、JSON 行    |


本工程发送侧曾用 HEX 定长风：`0xFF` + 4 字节 + `0xFE`（`Serial_SendPacket`）；接收侧当前示例是 **文本可变长**：`@` 开头、`\r\n` 结尾。

### 5.3 HEX 数据包 vs 文本数据包


|      | HEX（二进制）包                  | 文本包                        |
| ---- | -------------------------- | -------------------------- |
| 内容   | 原始字节，如 `FF 01 02 03 04 FE` | 可读字符，如 `@LED_ON\r\n`       |
| 调试   | 串口助手选 **十六进制显示/发送**        | 直接当字符串看                    |
| 解析   | 按数值比较（`RxData == 0xFF`）    | 按字符（`RxData == '@'`）再拼成字符串 |
| 典型用途 | 紧凑控制、传感器透传                 | AT 指令、日志、人读命令              |


二者可以并存：底层都是字节；区别在 **约定字节含义是数值还是 ASCII 字符**。

### 5.4 状态机是什么

**状态机（State Machine）**：程序在任一时刻处于有限个 **状态** 之一；收到事件（这里是「新到一字节」）后，按规则 **转移状态** 并可能存数据。

适合串口收包：字节是 **一个一个到的**，不能假设「一次中断就来整包」。

```text
        收到包头              收满/收到包尾
  WAIT ────────► RECEIVING ──────────────► DONE
   ▲                 │                       │
   └─────────────────┴── 出错/超时（可选）───┘
```

### 5.5 用状态机接收数据包

**思路**：在 `USARTx_IRQHandler` 里每收到 1 字节，只做「当前状态下该干什么」，收完置 `RxFlag`，主循环再处理。

#### 固定包长（HEX）示意

协议：`0xFF` + 4 字节载荷 + `0xFE`


| 状态  | 含义  | 收到字节后                              |
| --- | --- | ---------------------------------- |
| 0   | 等包头 | `== 0xFF` → 状态 1，下标清 0             |
| 1   | 收载荷 | 写入缓冲区，满 4 字节 → 状态 2                |
| 2   | 等包尾 | `== 0xFE` → 置 RxFlag，回状态 0；否则错误回 0 |


#### 可变包长（文本）——与本工程一致

协议：`@` + 内容 + `\r` + `\n`（工程 [Serial.c](../project/Hardware/Serial.c)）


| 状态 `RxState` | 含义  | 行为                                             |
| ------------ | --- | ---------------------------------------------- |
| **0**        | 等包头 | 收到 `'@'` 且上次包已处理完 → 进状态 1                      |
| **1**        | 收内容 | 普通字符写入 `Serial_RxPacket[]`；收到 `'\r'` → 进状态 2   |
| **2**        | 等换行 | 收到 `'\n'` → 加 `'\0'`、置 `Serial_RxFlag=1`，回状态 0 |


```text
状态 0 ──收到 '@'──► 状态 1 ──收到 '\r'──► 状态 2 ──收到 '\n'──► 状态 0
                        │
                        └── 其它字符写入缓冲区
```

对应代码骨架（与工程同结构）：

```c
void USART1_IRQHandler(void)
{
    static uint8_t RxState = 0;
    static uint8_t pRxPacket = 0;

    if (USART_GetITStatus(USART1, USART_IT_RXNE) == SET)
    {
        uint8_t RxData = USART_ReceiveData(USART1);

        if (RxState == 0)
        {
            if (RxData == '@' && Serial_RxFlag == 0)
            {
                RxState = 1;
                pRxPacket = 0;
            }
        }
        else if (RxState == 1)
        {
            if (RxData == '\r')
                RxState = 2;
            else
                Serial_RxPacket[pRxPacket++] = RxData;
        }
        else if (RxState == 2)
        {
            if (RxData == '\n')
            {
                RxState = 0;
                Serial_RxPacket[pRxPacket] = '\0';
                Serial_RxFlag = 1;   /* 主循环里用完后清 0 */
            }
        }
        USART_ClearITPendingBit(USART1, USART_IT_RXNE);
    }
}
```

**使用注意**：

1. ISR 里只拼包、置标志；解析命令放主循环。
2. `Serial_RxFlag == 1` 时勿覆盖缓冲区（工程用 Flag 挡下一包）。
3. 可变长要防 `pRxPacket` 越界（加长度上限）。
4. 文本包别忘了字符串结尾 `'\0'`。

## 六、知识怎么串起来

```text
通信
 └─ 协议：8N1、包头包尾、命令含义
 └─ 接口：TTL（入门）/ RS-232 / RS-485
        └─ USART 硬件：按比特时序收发字节
               └─ 软件状态机：字节 → 完整数据包 → 主循环执行
```


| 你想做的事   | 落到哪一章                   |
| ------- | ----------------------- |
| 和电脑对日志  | TTL + USART + 文本        |
| 搞清乱码    | 时序：波特率/接线交叉             |
| 发开关灯命令  | 数据包 + 状态机               |
| 以后上现场总线 | RS-485 + 应用协议（如 Modbus） |

## 七、常见问题


| 现象           | 可能原因                   | 处理                        |
| ------------ | ---------------------- | ------------------------- |
| 串口无输出        | TX/RX 未交叉；未共地；未初始化     | TX↔RX；接 GND；查 `USART_Cmd` |
| 乱码           | 波特率不一致                 | 双方统一 9600 或 115200        |
| 只能发不能收       | 未开 RXNE / NVIC；RX 脚模式错 | 查中断与 GPIO                 |
| 状态机收不全       | 包尾约定与发送端不一致            | 对一下是 `\r\n` 还是只要 `\n`     |
| 第一包之后卡死      | `RxFlag` 未清，一直挡在状态 0   | 主循环处理完清 Flag              |
| 接 RS-232 口烧脚 | 电平不匹配                  | 加 MAX232 等，勿直连            |

  → **TTL（3.3V）**。
3. USB 口、COM 口、USART 各是什么？
  → USB 口：电脑物理接口；COM 口：OS 里的虚拟串口名；USART：MCU 硬件串口外设。
4. 异步一帧里，谁拉低表示「开始」？
  → **起始位**。
5. `USART_ITConfig` 里的 IT 是什么缩写？
  → **Interrupt（中断）**。
6. 固定包长和可变包长如何判断「一包结束」？
  → 定长：收满 N；变长：包尾或长度字段。
7. 本工程文本包三个状态分别等什么？
  → `'@'` → 内容 → `'\r'` → `'\n'`。
